Decision Rules

Non-Negotiable Rules

  • Never log, print, embed, or commit secrets. Not in generated code, docs, prompts, or browser consoles.
  • Read-only credentials for discovery. Escalate to write only at deploy time after approval.
  • Encrypt in transit. All broker connections use tcps:// or wss:// in production.
  • Inject secrets via the platform. Kubernetes Secrets, Vault, or secure env vars. Never bake into images.
  • Scope credentials minimally. Dedicated ACL profile, client profile, and username per integration.
  • Validate TLS certificates. SSL_VALIDATE_CERTIFICATE=true always.

Per-Integration Broker Security

Every micro-integration gets three dedicated broker objects:

ObjectPurposeKey Setting
ACL ProfileControls publish/subscribe scopepublishTopicDefaultAction: disallow
Client ProfileControls resource limitsConnection, flow, and subscription limits
Client UsernameBinds ACL + profile to identityUnique per integration, strong password

Core Principle

Design events for the organization, not for the integration. A well-designed event is consumable by any future application without changes to the producer.

Search Before Creating

  • Event Portal — does an event with this business meaning already exist?
  • Schema registry — does a schema for this entity exist?
  • Prior runs — check RUNS_ROOT for previous integrations in the same domain.
  • Existing connectors — check WORKSPACE_ROOT for reusable adapters.

Reuse Decision Matrix

SituationAction
Exact event + schema exists, RELEASEDConsume it. Add a consumer application version.
Event exists, schema needs extensionNew schema version (additive only), new event version.
Similar event in different domainEvaluate sharing. Mark shared: true if cross-domain.
No matching eventCreate new following the full registration order.

Schema Evolution

  • Always additive. New fields are optional with defaults.
  • Never remove or rename fields — create a new version.
  • Every schema includes: eventId, eventTimestamp, source, correlationId.
  • Use JSON Schema draft-07 or later.

Source-Agnostic Design

The factory is source-agnostic. The lifecycle is the same regardless of the input.

SourceDiscoveryConnector Pattern
OpenAPI / RESTPaths, operations, schemas, authHTTP ingress, polling, or webhook
DatabaseTables, FKs, timestamps, CDC capabilityCDC (preferred), polling, outbox
Webhook / SaaSEvent catalog, signatures, retriesHTTP receiver, HMAC verification
Message QueueTopics, formats, consumer groupsConsumer bridge to Solace
File / Object StorePatterns, schemas, arrival frequencyFile watcher, record extraction
Custom / ExistingSource code, config, tests, qualityReuse, adapt, or replace

Supported Databases

Any JDBC-compatible database: PostgreSQL, MySQL, Oracle, Microsoft SQL Server, and more. CDC strategy is inferred from the database capabilities (logical replication, binlog, LogMiner, Change Tracking).

Wildcard Subscriptions

PatternMatches
*Exactly one topic level
>One or more remaining levels (must be last)
acmeretail/orders/order/*/v1/>All v1 order events, any action/region/ID
acmeretail/orders/>All order-related events

Anti-Patterns

  • Do NOT put environment names in topics — use separate VPNs
  • Do NOT embed trace IDs — use Solace-Correlation-ID header
  • Do NOT use special characters (# _ * > ! spaces)
  • Do NOT put high-cardinality keys first — put them last

Acceptance Standard

A run is only complete when:

  • Source understood — discovery report with entities and risks
  • Events designed for reuse — schemas with metadata, versioned, registered
  • Integration generated — compilable project with tests and Dockerfile
  • Tests passed — unit, contract, and happy-path publish
  • Deployment live — health check passing, image tag recorded
  • Events flowing — message on Solace topic with correlation ID
  • Event Portal complete — full design graph linked
  • Security hardened — dedicated ACL, TLS, no embedded secrets
  • Documentation written — operator runbook, event catalog, consumer guide
  • Operations handoff — monitoring, alerting, dead-letter, maintenance plan
